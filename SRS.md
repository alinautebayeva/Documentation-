# Software Requirement Sepicfication для продукта MeetRoom

## 1. Введение
### 1.1 Цель документа


Цель этого SRS (Software Requirements Specification) — предоставить исчерпывающее описание требований для разработки продукта MeetRoom, чтобы обеспечить понимание функциональности, задач и ожиданий для всех участников команды разработки и других заинтересованных сторон.

### 1.2 Описание продукта 
__MeetRoom__ — это мобильное приложение для бронирования переговорных комнат в офисе, предназначенное для сотрудников компании. Продукт позволяет быстро и удобно забронировать переговорную комнату, а также сокращает количество двойных бронирований. (далее по тексту - *продукт*, *приложение*)

### 1.3 Заинтересованные стороны
- Бизнес-заказчик — TechSolutions Ltd. 
- Команда разработки — разработчики, тестировщики, аналитики, дизайнеры, менеджеры, лица, ответственные за поддержку продукта. 

## 2. Бизнес-требования
### 2.1 Основные цели продукта
На стадии запуска внедрение приложения MeetRoom направлено на достижение следующих бизнес-целей:

1. Повышение эффективности использования переговорных комнат: сотрудники смогут видеть занятость комнат в реальном времени, что снизит количество простоев и неиспользуемых ресурсов;

2. Снижение конфликтов и дублирующих бронирований:  автоматическая проверка занятости и подтверждение бронирования в приложении минимизирует вероятность пересечений и двойных бронирований;

3. Сокращение временных затрат сотрудников: сотрудники смогут быстро находить свободные комнаты и создавать бронирования через мобильное приложение;

4. Повышение прозрачности и контроля для руководства и администрации: система предоставляет руководству статистику по использованию переговорных комнат.

### 2.2 Ключевые показатели эффективности

#### 1. Процент успешных бронирований:

- Описание: доля бронирований, которые прошли без конфликтов и были подтверждены системой. 
- Целевое значение: ≥ 95% всех бронирований.

#### 2. Снижение количества конфликтов бронирований

- Описание: количество случаев двойного бронирования переговорной комнаты по сравнению с периодом до внедрения приложения. 
- Целевое значение: снижение на ≥ 70%.

#### 3. Время создания бронирования

- Описание: среднее время, затрачиваемое сотрудником на поиск свободной комнаты и создание бронирования через приложение. 
- Целевое значение: ≤ 2 минут на бронирование.

#### 4. Использование приложения сотрудниками

- Описание: доля сотрудников, активно использующих приложение для бронирования переговорок.
- Целевое значение: ≥ 80% сотрудников офиса в первые 3 месяца после внедрения.

#### 5. Количество пропущенных или отмененных встреч по причине недоступности переговорок

- Описание: количество встреч, которые не состоялись из-за отсутствия информации о доступных комнатах.
- Целевое значение: снижение на ≥ 60% по сравнению с периодом до запуска приложения.

#### 6.Время отклика приложения

- Описание: среднее время отклика системы при создании, изменении или отмене бронирования.
- Целевое значение: ≤ 3 секунд для 95% операций.

#### 7. Удовлетворенность пользователей

- Описание: оценка удобства приложения среди сотрудников по результатам опроса (например, 5-балльная шкала).
- Целевое значение: ≥ 4 баллов.

## 3. Требования к внешним интерфейсам
Требований к внешним интерфейсам на данном этапе не предъявляется.

## 4. Функциональные требования 

### 1. Авторизация и аутентификация

__FR-1.1__
Система должна предоставлять пользователю возможность входа в мобильное приложение с использованием корпоративной учетной записи (email и пароль).

__FR-1.2__
Система должна запрещать доступ к функционалу приложения неавторизованным пользователям.

__FR-1.3__
Система должна сохранять сессию пользователя до момента выхода из аккаунта или истечения срока действия сессии.

***User Story***

> Как сотрудник компании,
я хочу войти в приложение под своей корпоративной учетной записью,
чтобы иметь доступ к бронированию переговорных комнат.

### 2. Просмотр списка переговорных комнат

__FR-2.1__
Система должна отображать список всех доступных переговорных комнат.

__FR-2.2__
Для каждой переговорной комнаты система должна отображать:

- название комнаты;
- этаж;
- вместимость;
- список оборудования;
- текущий статус (свободна / занята).

__FR-2.3__
Система должна предоставлять возможность фильтрации переговорных комнат по:

- вместимости;
- наличию оборудования;
- этажу.

***User Story***

> Как сотрудник,
я хочу видеть список переговорных комнат с их характеристиками,
чтобы быстро выбрать подходящую комнату для встречи.

### 3. Просмотр доступности переговорной комнаты

__FR-3.1__
Система должна отображать доступность выбранной переговорной комнаты на заданную дату и время.

__FR-3.2__
Система должна предотвращать бронирование переговорной комнаты, если она уже занята в выбранный временной интервал.

***User Story***

>Как сотрудник,
я хочу проверять доступность переговорной комнаты на нужное время,
чтобы избежать конфликтов при бронировании.

### 4. Создание бронирования

__FR-4.1__
Система должна предоставлять возможность создать бронирование переговорной комнаты с указанием:

- даты и времени начала;
- даты и времени окончания;
- выбранной комнаты;
- списка участников встречи.

__FR-4.2__
После успешного создания бронирования система должна сохранять его и отображать в списке бронирований пользователя.

__FR-4.3__
Система должна синхронизировать созданное бронирование с корпоративным календарем.

***User Story***

> Как сотрудник,
я хочу забронировать переговорную комнату на определенное время,
чтобы провести встречу без необходимости искать свободное помещение.

### 5. Изменение и отмена бронирования

__FR-5.1__
Система должна позволять пользователю изменять параметры своего бронирования до начала встречи.

__FR-5.2__
Система должна позволять пользователю отменять свое бронирование.

__FR-5.3__
При изменении или отмене бронирования система должна обновлять данные в корпоративном календаре.

***User Story***

> Как сотрудник,
я хочу изменять или отменять свое бронирование,
чтобы адаптироваться к изменениям в расписании.

### 6. Просмотр истории бронирований

__FR-6.1__
Система должна отображать пользователю список его активных, завершенных и отмененных бронирований.

__FR-6.2__
Система должна предоставлять возможность просмотра деталей бронирования.

***User Story***

>Как сотрудник,
я хочу видеть историю своих бронирований,
чтобы контролировать свои встречи и планировать будущие.

### 7. Уведомления

__FR-7.1__
Система должна отправлять push-уведомление пользователю за 10 минут до начала встречи.

__FR-7.2__
Система должна отправлять уведомление пользователю при изменении или отмене бронирования.

***User Story***

>Как сотрудник,
я хочу получать уведомления о предстоящих встречах,
чтобы не пропускать забронированные переговоры.

### 8. Обработка ошибок

__FR-8.1__
Система должна отображать пользователю сообщение об ошибке в случае невозможности создания бронирования.

__FR-8.2__
Система должна информировать пользователя о причинах отказа (например, комната занята).

***User Story***

> Как сотрудник,
я хочу получать понятные сообщения об ошибках,
чтобы понимать, почему бронирование не удалось.

## 5. Требования к Usability

### 5.1 Простота использования
Интерфейс: Максимально интуитивно понятный интерфейс, продуманные иконки и навигация.
Доступность функций: Часто используемые функции, такие как «Создать бронирование» и «Мои бронирования», должны находиться на видном месте и быть доступны в минимальное количество кликов.

### 5.2 Адаптивный дизайн
Отображение на устройствах: Удобное отображение интерфейса на экранах разных размеров — мобильные телефоны, планшеты.

### 5.3 Визуальная поддержка пользователя.
Обратная связь: Должна отображаться при каждом действии пользователя (например, анимация загрузки, оповещение об успехе или ошибке при совершении действия).

### 5.4 Скорость взаимодействия
Оптимизация: Минимизация времени загрузки страниц, использование кэширования для повторного доступа к данным.
Ответ сервера: Время отклика не должно превышать 2 секунд для всех основных операций.

### 5.5 Доступность
Контраст и шрифты: Высококонтрастный текст, удобочитаемые шрифты и адаптивные элементы для взаимодействия.

## 6. Нефункциональные требования
### 1. Производительность

__NFR-1.1__
Система должна обеспечивать отклик на действия пользователя (просмотр списка переговорок, создание бронирования, изменение статуса) в течение не более 3 секунд для 95% операций.

__NFR-1.2__
Приложение должно корректно обрабатывать до 500 одновременных пользователей без деградации производительности.

### 2. Надежность и устойчивость

__NFR-2.1__
Приложение должно корректно обрабатывать временную недоступность корпоративного календаря, отображая соответствующее сообщение пользователю.

__NFR-2.2__
Система должна обеспечивать автоматическое восстановление соединения после кратковременного сбоя сети и синхронизировать изменения.

### 3. Доступность

__NFR-3.1__
Приложение должно быть доступно пользователям не менее 99% рабочего времени в рабочие дни.

__NFR-3.2__
Поддержка работы на платформах:

- Android ≥ 10
- iOS ≥ 14

### 4. Безопасность

__NFR-4.1__
Доступ к приложению разрешен только авторизованным сотрудникам компании через корпоративный логин.

__NFR-4.2__
Все данные пользователей и бронирований должны храниться и передаваться в зашифрованном виде (HTTPS / TLS 1.2+).

__NFR-4.3__
Приложение должно предотвращать несанкционированный доступ и обеспечивать автоматический выход пользователя при длительном отсутствии активности.
